# RU Voice & Narrative — камертон корпуса

> **Зачем этот документ.** Гейты `style-guide-check`, `check-rfc-modals`, `check-md-links`
> зелёные, а корпус читается как «набор бессвязных сведений». Причина: проверки
> измеряют поверхность (англицизмы, заголовки, ссылки), но слепы к связности
> аргумента, авторскому голосу и убедительности. Этот камертон задаёт ту ось,
> которую не ловит ни один скрипт. От него правится проза всех 15 глав.
>
> **Граница.** [`reference/06-ru-style-guide.md`](../reference/06-ru-style-guide.md) — про
> *слова* (что латиницей, RFC-модальности). Этот документ — про *голос и течение мысли*.
> Они не пересекаются и не конфликтуют.

## 0. Эталон, который у нас уже есть

Самый сильный текст проекта — [`core/renar-core.md`](../core/renar-core.md). У него есть
голос:

> «AI красиво формулирует требования и красиво пишет тесты, но реализация всё равно
> „не то“. Клиент при приёмке говорит: „мы это так не имели в виду“. Тесты зелёные,
> продукт неверный.»

Это эталон. Нормативный корпус этот голос потерял — задача вернуть его, **не теряя
нормативной строгости**. Мы пишем не легче, а яснее.

## 1. Главный критерий качества (новый контур)

Старый контур: «гейты зелёные». Новый контур, который добавляем:

> **Умный внешний читатель открывает главу, читает сверху вниз и выходит, понимая
> идею и веря ей — без прыжков по `§`-ссылкам и без перевода в голове.**

Это проверяется не скриптом, а чтением (человек или adversarial-агент). Эталонные
читаемые стандарты — ISO/IEEE и хорошие RFC: точные и при этом читаемые. Это
наш ориентир, а не «упростить».

## 2. Авторский голос — принципы

1. **Сначала смысл, потом формализм.** Абзац объясняет идею человеческими словами,
   и только затем приходят таблица, схема, frontmatter. Никогда не наоборот.
2. **Один уверенный автор, а не комитет.** Утверждаем прямо. Убираем защитную
   воду («следует отметить, что», «как было сказано выше», «в рамках данного»).
3. **Ритм.** Чередуем длинные и короткие предложения. После плотного абзаца —
   короткая фраза, которая ставит точку. (Образец — Core: «Тесты зелёные, продукт
   неверный.»)
4. **Конкретность вместо абстракции.** Где можно — пример, сценарий, цифра. «80%
   инцидентов в продакшене — это негативные сценарии» сильнее любого определения.
5. **Нормативность — это сила, а не сухость.** «обязан» / «запрещён» звучат твёрдо;
   вокруг них — живая проза, объясняющая *почему* норма именно такая.

## 3. Нарративный хребет всего корпуса

Корпус должен читаться как одна книга с одним аргументом, а не как 15 RFC. Сквозная
линия (каждая глава делает один шаг):

1. **Боль (00).** AI пишет код за минуты — узким местом становится корректность
   спецификации, а не кода. Требования дрейфуют между ТЗ, интерпретацией, тестами,
   реализацией.
2. **Идея (02).** Перевернуть источник истины: побеждает требование, не код.
3. **Опора (03, V1–V6).** Носитель обязан давать историю и provenance, иначе идея
   физически не работает — поэтому фундамент носителя заложен сразу после идеи.
4. **Договорный шов (07, ADAPT).** Сделать интерпретацию ТЗ явной и подписанной —
   ловить расхождение до кода.
5. **Структура истины (06, 08, 09).** Иерархия требований, спецификации, тесты
   парами — чтобы агент не «зеленил» счастливый путь.
6. **Движение (10).** Lifecycle и гейты обеспечивают enforcement.
7. **Печать (11, 13).** Уровни зрелости и заявление о соответствии замыкают круг.

> **Нумерация после реордера (вариант A).** Идея (02) и опора-носитель (03)
> сознательно вынесены вперёд: главы об артефактах (06–10) на них опираются. Спайн
> читается в порядке глав, кроме рокировки 06↔07 (иерархию вводим до договорного
> шва ADAPT — это осознанный риторический ход, не ошибка нумерации).

Каждая глава должна *знать своё место в этой цепочке* и явно подхватывать предыдущий
шаг в первом абзаце — прозой, а не таблицей.

## 4. Запрещённые приёмы (источники «бессвязности»)

| Приём | Почему плохо | Чем заменить |
|---|---|---|
| Формульная открывашка «Назначение главы / кому-зачем-куда» | Выдаёт машинную сборку; одинаковый каркас на каждой главе | Живой первый абзац: подхват предыдущего шага + что здесь решается |
| Таблица «Связь с главой X» в конце как единственный мост | Это навигация, выданная за переход мысли | Прозовый мостик в теле + ссылки inline там, где они реально нужны |
| `§`-ссылки в скобках почти в каждом предложении | Текст превращается в гипертекстовый индекс | Ссылку — только когда читатель реально пойдёт туда; остальное в один блок «Перекрёстные ссылки» |
| frontmatter / схема до объяснения | Читатель видит YAML раньше, чем понял зачем | Сначала абзац-смысл, потом схема |
| Кластер латиницы в русском предложении | «Перевод в голове» | См. §5 |

## 5. Де-макаронизация — по читаемости, а не по счётчику

**Остаётся латиницей** (не трогаем): canonical-идентификаторы и closed-list — `BR`,
`SR`, `TR`, `ADAPT`, `SPEC-*`, `TC`, `QG-0..QG-4`, `V1–V6`, lifecycle-статусы
(`draft`, `approved`, `verified`, …), `MVR`, уровни `RENAR-1..5`.

**Переводим** (рабочий словарь — пополняется):

| Было | Стало |
|---|---|
| derived (артефакт) | производный |
| interchangeable | взаимозаменяемый |
| reverse-engineering | обратное восстановление (требований из кода) — пояснить при первом упоминании |
| vendor-нативный / vendor-native | привязанный к конкретному инструменту |
| WIP | черновик / работа в процессе |
| continuous reconciliation | непрерывная сверка |
| throw spec over the wall | «перебросить спецификацию через забор» (как образ) |
| first-class artifact | полноценный артефакт |
| position statement | позиционное утверждение |
| concept-level | на уровне идеи / обзорно |
| formal change procedure | формальная процедура изменения (стандарта) |
| research draft | исследовательский черновик |
| public review | публичное обсуждение |
| version bump (minor/major) | повышение версии (minor-/major-) |
| migration guidance | руководство по миграции |
| loss-of-conformance | потеря соответствия |
| self-assessment checklist | чек-лист самооценки |
| corresponding alignment | согласующая правка |
| project-local | локальный для проекта / локально на уровне проекта |

**Принятые термины международной RE/AI-практики** оставляем, но определяем по-русски
при первом упоминании: `Spec-Driven Development (SDD)` → «разработка от спецификации
(SDD)»; `traceability` → «прослеживаемость (traceability)»; `provenance` →
«происхождение / provenance»; `drift` → «дрейф».

## 6. Что НЕ трогаем (граница нормативности)

- RFC-2119 модальности: `обязан / должен / следует / рекомендуется / может /
  запрещён` — нельзя менять *уровень* (только синоним того же уровня). Гейт
  `check-rfc-modals`.
- Mandatory clauses, MVR-1..7, closed lists — текст можно сделать читаемее, но
  состав и нормативную силу не меняем.
- Canonical-идентификаторы в frontmatter, YAML, именах файлов — referential
  integrity со схемами и hooks.
- `research/*` со статусом `frozen` — не редактируем.

## 7. Решённый вопрос: «носитель», не `substrate`

**Решение (сессия #27, утверждено пользователем):** канонический термин —
**«носитель»** (склоняется по падежам). Латиницей `substrate` остаётся только в именах
полей (`substrate-capabilities`), коде/YAML, именах файлов (`03-substrate-versioning.md`)
и англоязычных glossах возможностей. Это снимает разнобой, на который раньше указывал
этот параграф: Core и нормативный корпус теперь зовут концепт одинаково.

Компаунд-маппинг: `substrate-agnostic` → «независимый от носителя»; `substrate-specific`
→ «специфичный для носителя»; `substrate-native` → «нативный для носителя»;
`cross-substrate` → «межносительный / сквозной»; `substrate-A/B` → «носитель A/B»;
`requirements substrate` → «носитель требований». Затрагивает §0.8, reference/06
(Bucket A) и §4-terms — обновляются в рамках конверсии главами.
