# ADR-010: Граница «descriptive discovery vs agent-протокол»

> **Статус: proposed (2026-06-28).** Фиксирует, какие сигналы агентной
> дискаверабилити допустимо публиковать на renar.tech, не нарушая принцип
> **«RENAR = стандарт, НЕ реализация»** (CLAUDE.md NEVER BREAK). Реализация
> волны C сайта (Agent Card) опирается на рекомендацию этого ADR.

## Контекст

Аудит renar.tech по сканеру [isitagentready.com](https://isitagentready.com)
(сессия #42) показал: базовый слой агентной дискаверабилити закрыт волнами A+B
(robots с AI-bot-правилами и Content-Signal, llms.txt + llms-full.txt, JSON-LD,
markdown content-negotiation, Link-заголовки). Открытой осталась категория
**Protocol Discovery**: Agent Card, MCP Server Card, WebMCP, OAuth-discovery,
а также **DNS-AID** (`_agent.<домен>` TXT-запись) из категории Discoverability.

Эти сигналы неоднородны по природе. Часть из них — **описательные метаданные**
(машиночитаемый паспорт ресурса), часть — **заявка на живой агентный сервис**
(endpoint, к которому клиент реально подключается по протоколу). Принцип
«RENAR = стандарт, НЕ реализация» проводит границу: реализация — это TAUSIK
(skill `req`) / KAI, а не репозиторий стандарта.

## Решение: две категории сигналов

**Категория 1 — Descriptive discovery (концерн сайта, ПУБЛИКУЕМ).**
Статические метаданные, описывающие *что* такое RENAR и *где* лежит корпус.
Они ничего не «исполняют», а лишь помогают агенту найти и понять документ.
Сюда уже отнесены `robots.txt`, `llms.txt`, `llms-full.txt`, JSON-LD,
markdown content-negotiation. К этой же категории относится **Agent Card** —
если он оформлен как *описательный паспорт ресурса* (имя, описание, ссылки на
корпус и схемы, провайдер, лицензия), без объявления живого RPC-endpoint.
Публикация такого паспорта эквивалентна публикации `llms.txt` и **не**
является реализацией RENAR.

**Категория 2 — Live agent protocol (реализация, НЕ публикуем из этого репозитория).**
Сигналы, которые объявляют работающий агентный сервис, к которому клиент
подключается: **MCP Server Card / WebMCP** (требуют запущенного MCP-сервера),
A2A endpoint, OAuth Protected Resource. Это — реализация (TAUSIK/KAI), она
живёт вне репозитория стандарта и появляется только после соответствующего
решения. Соответствует уже заблокированной задаче `aiready-mcp-card` (C2).

## Рекомендации по задачам волны C

- **C1 `aiready-agent-card` — ПУБЛИКОВАТЬ** как descriptive Agent Card
  (`/.well-known/agent.json`): паспорт ресурса со ссылками на llms.txt,
  llms-full.txt, markdown-зеркало `/md/`, схемы, GitHub; провайдер Kibertum;
  лицензия CC BY-SA 4.0. **Без** объявления живого `url`/`endpoint`
  агентного RPC. Требует исключения `/.well-known/` из nginx-правила,
  блокирующего скрытые файлы (`location ~ /\.` → deny).

- **C2 `aiready-mcp-card` — ОТЛОЖИТЬ** (остаётся blocked). MCP Server Card
  публикуется только когда поднят реальный MCP-сервер (TAUSIK/KAI), отдающий
  стандарт по запросу. Это отдельное решение о реализации.

## Граничный случай: чего НЕ делаем

**Запрещено** публиковать «полноценный» A2A Agent Card или MCP Server Card,
объявляющий RPC-endpoint (skills с вызываемыми методами, `url` сервиса),
которого физически нет. Это (а) нечестно — клиенты сломаются при подключении;
(б) технически невалидно — сканер/клиенты при probe получат ошибку соединения;
(в) нарушает границу «стандарт ≠ реализация». Descriptive-паспорт без живого
endpoint — допустим; фейковый сервис — нет.

## DNS-AID — отдельная проблема (требует действия пользователя)

Спецификация **AID v2** ([aid.agentcommunity.org](https://aid.agentcommunity.org/docs/specification))
для записи `_agent.<домен>` (TXT) делает обязательными три поля: `v`, `u`
(endpoint агента), `p` (протокол: `mcp`/`a2a`/`openapi`/…). **Профиля
«только документация» в спецификации нет** — валидная AID-запись по определению
указывает на *живой агентный endpoint* (Категория 2).

Следствие: опубликованная AID-запись, чей `u=` ведёт на несуществующий адрес
(например `https://renar.tech/mcp` при отсутствующем MCP-сервере), —
**dangling**: AID-совместимые клиенты при обнаружении записи попытаются
подключиться и получат ошибку. Это хуже, чем отсутствие записи.

**Рекомендация:** до появления реального агентного endpoint (C2) запись
`_agent.renar.tech` следует **снять** (или не публиковать). Когда MCP-сервер
TAUSIK/KAI будет поднят — вернуть запись вида
`v=aid2;u=https://renar.tech/mcp;p=mcp;s=…;d=https://renar.tech/llms.txt`.
Альтернатива «оставить dangling-запись ради галочки сканера» отвергается по
тем же причинам, что и фейковый Agent Card.

## Последствия

- Волна C сайта сводится к одному честному артефакту (C1, descriptive Agent
  Card) + nginx-правка; C2 (MCP) и живой AID-endpoint ждут реализации.
- Нормативный корпус не затрагивается; bump версии стандарта не требуется.
- Мета-вопрос (вне этого ADR): должен ли *сам корпус* RENAR рекомендовать
  агентную дискаверабилити для репозиториев требований — это контент стандарта,
  решается отдельно (потенциальное приложение).

## Resolution (2026-06-28): валидная AID-запись через `p=openapi` над статикой

Пользователь опубликовал запись `_agent.renar.tech` с `u=https://renar.tech/mcp;p=mcp`,
но MCP-сервера нет → запись dangling (раздел DNS-AID выше). Поскольку MCP не делаем,
а профиля «только документация» в AID нет, выбран **третий путь**: указать `u=` на
**реальный, уже работающий** read-only ресурс через протокол `openapi`.

Опубликован статический **OpenAPI 3.1** (`https://renar.tech/openapi.json`),
описывающий существующие GET-эндпоинты сайта: `/llms.txt`, `/llms-full.txt`,
`/md/{section}/{chapter}.md`, `/.well-known/agent.json`. Все они реально отдают
заявленный content-type — документ не описывает несуществующих или write-операций.
Это статическая раздача файлов, а не серверная логика, поэтому остаётся в
**Категории 1** (descriptive discovery): живого агентного RPC-сервиса не вводится.

Итоговая корректная AID-запись (заменяет dangling-вариант):

```
_agent.renar.tech.  IN  TXT  "v=aid2;u=https://renar.tech/openapi.json;p=openapi;s=RENAR requirements-engineering standard;d=https://renar.tech/llms.txt"
```

`p=mcp;u=…/mcp` возвращается позже — когда (и если) поднимут MCP-сервер TAUSIK/KAI (C2).

## Открытый вопрос партнёру

Согласен ли партнёр с классификацией «descriptive Agent Card = концерн сайта,
не реализация»? Если партнёр сочтёт даже описательный паспорт частью
реализации — C1 тоже уходит в TAUSIK/KAI, и сайт остаётся на волнах A+B.
