# Style Guide для AI-генерации REQ-артефактов

> Версия: черновик 0.1 | Дата: 2026-05-03
> Назначение: стиль, тон, структура, длина для AI-агентов, генерирующих REQ-артефакты (BR, SR, UIC, AIC, TC, Impact Analysis). Снимает класс рисков «разные модели → разный стиль → drift восприятия».

---

## 1. Не дублирует SENAR

SENAR §5 (Agent Instrumentation) и SENAR Guide §4 (Working with AI agents) описывают **общий контракт** взаимодействия с AI-агентами. Этот документ — **узкий style guide для AI, генерирующего артефакты требований**.

Аналог в индустрии: Google Developer Documentation Style Guide для технических писателей, Microsoft Style Guide for technical writing, Diátaxis framework. У всех — нормирован стиль для документации, чтобы независимо от автора tone оставался ровным.

---

## 2. Принципы

### 2.1 Стиль = договорная точность, не художественная литература

REQ-артефакт — **договорной документ** между клиентом, командой и системой. Никаких метафор, никакого «красочного» языка, никаких ассоциаций. Только однозначные утверждения.

Хорошо:
> Система должна принимать заявку на регистрацию через POST /auth/register.

Плохо:
> Пользователи смогут легко и удобно создавать аккаунты через современный API.

### 2.2 Один артефакт = одна мысль

Каждый BR, SR, TC покрывает **одну** концепцию. Если AI-агент склонен «упаковать» 3 темы в один SR — он должен декомпозировать, не упаковывать.

Сигнал упаковки: `И` в title, multiple actions в одном утверждении.

Плохо:
> SR-05: Регистрация, авторизация и восстановление пароля

Хорошо:
> SR-05: Регистрация клиента
> SR-06: Авторизация клиента
> SR-07: Восстановление пароля

### 2.3 Нет претензий на полноту

AI не должен «дорабатывать» требование от себя, если в источнике (ТЗ) этого нет. Каждое утверждение либо имеет citation, либо помечено `derived` с явным обоснованием (см. [02-agent-driven-principles.md Принцип 3](02-agent-driven-principles.md)).

Плохо:
```markdown
- При регистрации проверяется уникальность email.
- При регистрации проверяется уникальность телефона.   ← в ТЗ нет, AI домыслил
```

Хорошо:
```markdown
- При регистрации проверяется уникальность email. [TZ-2026-001 §4 ФТ-001 line 142]
```

Если AI считает, что нужна проверка телефона — он не пишет её в SR, а добавляет в Critic-output: «возможно нужна проверка уникальности телефона — нет в ТЗ, требует уточнения у stakeholder».

### 2.4 Однозначность важнее красоты

Если выбор между точным длинным предложением и компактным неоднозначным — AI выбирает точное длинное. Длина — не enemy, ambiguity — enemy.

## 3. Структура и длина

### 3.1 BR — Business Requirement

**Body** (без frontmatter): 30-80 строк.

**Обязательные секции**:
- `## Потребность` — одно предложение по шаблону `[Роль] должен [действие], чтобы [бизнес-цель].`
- `## Критерии успеха` — 3-7 измеримых пунктов.
- `## Контекст` — 5-15 строк.

**Запрещено**:
- Технологии («через REST API», «Postgres», «React»).
- Конкретные экраны или поля БД.
- Размышления автора («мы думаем», «возможно», «было бы хорошо»).

**Тон**: формальный, императивный («должен», не «может»).

### 3.2 SR — System Requirement

**Body**: 40-150 строк.

**Обязательные секции**:
- `## Требование` — одно предложение по шаблону `Система должна [поведение]. [Условие].`
- `## Поведение` — 5-20 пунктов с inline citations.
- `## Ограничения` — 3-10 пунктов.

**Опциональные секции** (когда применимо):
- `## Не входит в это требование` — explicit out-of-scope.
- `## Связанные SR` — cross-links.

**Запрещено**:
- Названия таблиц БД, конкретных функций, классов.
- Frameworks («через FastAPI», «React-компонент»).
- Конкретные UI-paths («кнопка в правом углу») — это в UIC.

**Тон**: формальный, точный, поведенческий («система отвечает 422», не «возвращается ошибка»).

### 3.3 UIC — UI Concept

**Body**: 80-300 строк.

**Обязательные секции**:
- `## Общая структура интерфейса` — навигация, layout.
- `## Ключевые экраны` — каждый экран отдельным sub-блоком.
- `## Пользовательские сценарии` — 3-7 сценариев в нарративе.
- `## Тон и стиль`.
- `## Сквозные элементы`.

**Опциональные**:
- `## Ссылки на макеты` (Figma/Miro).
- `## Примечания` (отклонённые решения).

**Тон**: пользователь-центричный нарратив с поведенческой точностью. «Менеджер видит ленту заказов, нажимает на заказ, открывается детальный экран», не «UI должен иметь listing component».

### 3.4 AIC — AI Concept

**Body**: 50-200 строк.

**Обязательные секции**:
- `## Концепт` — описание архитектурного подхода.
- `## Ключевые решения` — 3-7 пунктов в формате `[выбор] — [обоснование]`.
- `## Альтернативы, отклонённые` — что рассматривали, почему отказались.

**Дополнительно** (для RENAR-5):
- `## AI-параметры` — модели, температуры, baseline-датасет.
- `## Метрики успеха` — link на eval-тесты.

**Тон**: технико-аналитический. Здесь допустимо описание технологий — это AI architecture document.

### 3.5 TC — Test Case

**Body**: 30-80 строк.

**Обязательные секции**:
- `## Контекст` — 2-5 строк.
- `## Предусловия` — 3-10 пунктов.
- `## Шаги` — 3-15 пунктов.
- `## Критерий успеха (Pass)` — бинарный, наблюдаемый, воспроизводимый.
- `## Критерий неуспеха (Fail)` — список наблюдаемых признаков нарушения.
- `## Постусловия`.
- `## Не входит в этот тест` — out-of-scope с pointer на покрывающие TC.

**Тон**: исполнительный, четкий («выполняется X», «возвращается Y»).

### 3.6 Impact Analysis

**Body**: 50-200 строк (зависит от размера дельты).

**Обязательные секции**:
- `## Затронутые требования` — таблица с типом изменения.
- `## Затронутые TC` — таблица с действием.
- `## Затронутые задачи в backlog`.
- `## Open questions` — что нужно уточнить со stakeholder.

**Тон**: аналитический, без оценок. Не «это плохое изменение», а «это изменение затрагивает X требований и Y задач».

## 4. Лексикон

### 4.1 Использовать (canonical)

- «Система должна...» (модальность)
- «должен» / «не должен» (RFC 2119 vocabulary).
- «возвращает» (HTTP/API behavior)
- «предусловия» / «постусловия» (формальные термины тестирования)
- «требование» / «утверждение» / «assertion»
- «критерий приёмки» (acceptance criterion)

### 4.2 Не использовать

| Слово/фраза | Заменить на | Почему |
|---|---|---|
| «Хочется чтобы» | «Система должна» | Modal verb должен быть строгим |
| «Удобно» | конкретный измеримый критерий | Subjective |
| «Современный» | конкретные tech requirements в SR/AIC | Marketing language |
| «Качественный» | измеримые quality characteristics из ISO 25010 | Subjective |
| «Быстрый» | p95/p99 latency с числом | Vague |
| «Надёжный» | uptime/reliability с числом | Vague |
| «Возможно» / «Может быть» | (никогда в требованиях) | Hedging |
| «Скорее всего» | (никогда) | Same |
| «Желательно» | priority should/could во frontmatter | Mixing channels |
| «Понятный» | criteria for clarity (e.g. Flesch reading ease) | Subjective |
| «Гладкий experience» | UX criteria в UIC | Marketing |
| «Бесшовный» | конкретный технический критерий | Marketing |

### 4.3 Use Cases vs Stories vs Requirements

REQ использует **только canonical терминологию из [05-glossary-terminology.md](05-glossary-terminology.md)**. AI-агент **не имеет права** называть требование «User Story», «Use Case», «Scenario», «Capability» — это запрещённые синонимы.

## 5. Структура утверждений

### 5.1 Pass-критерий (TC)

Шаблон:
> `<actor>` `<action>` `<input>`. `<observable>` `<measurement>`.

Пример:
> POST /auth/login с body `{"email":"x@y.z", "password":"correct"}`. Response status = 200, body содержит JWT с `exp = now + 24h ± 1m`.

### 5.2 Fail-критерий (TC)

Шаблон: список наблюдаемых признаков нарушения, **включая** возможные побочные эффекты.

Пример:
> - Response status ≠ 200 (ошибка авторизации).
> - В 401-ответе указано конкретное поле, которое неверно (утечка информации).
> - В системный лог записывается plaintext password (security violation).
> - Email с уведомлением о входе НЕ отправляется при успешной авторизации (missing side effect).

Не допускается: «Pass не выполняется». Это не Fail-критерий, это negation.

### 5.3 SR-утверждения

Шаблон: `<actor>` `<action>` `<condition>`. `<consequence>`.

Пример:
> Система при первом запуске проверяет наличие разрешения NotificationListenerService. Если разрешение не выдано — отображается экран онбординга с кнопкой «Выдать доступ».

### 5.4 BR-утверждения

Шаблон: `[Роль]` должен `[действие]`, чтобы `[бизнес-цель]`.

Пример:
> Пользователь должен автоматически получать и хранить уведомления выбранных приложений в фоновом режиме, чтобы не пропустить важные сообщения.

## 6. Цитирование source

### 6.1 Inline citation

Каждое утверждение в BR/SR имеет inline-citation в квадратных скобках:

> При первом запуске отображается экран онбординга. [TZ-2026-001 §4 ФТ-001]

Формат: `[<TZ-id> <section> <optional: line>]`.

### 6.2 Derived маркер

Если утверждение не цитата из ТЗ, а логически выведено:

> Кнопка `Назад` блокируется на экране онбординга. [TZ-2026-001 §4 ФТ-001, derived: ТЗ требует «нельзя пропустить онбординг» → блокировка системного `back`]

Объяснение `derived` обязательно.

### 6.3 Multi-source

Если утверждение поддерживается несколькими источниками:

> Пароль хранится в зашифрованном виде. [TZ-2026-001 §6 НФТ-002, ISO 27001 A.10.1.1]

## 7. Формат AI-промптов для генерации (system prompt)

Каждый AI-агент, генерирующий REQ-артефакт, получает system prompt, состоящий из:

1. **Role**: «Ты архитектор требований по RENARу».
2. **Style guide reference**: link на этот документ.
3. **Glossary reference**: link на [05](05-glossary-terminology.md).
4. **Constraints**:
   - Каждое утверждение либо с citation, либо `derived` с объяснением.
   - Использовать только canonical термины.
   - Структура и длина согласно §3 этого документа.
   - Никаких запрещённых слов из §4.2.
5. **Output schema**: точный YAML frontmatter format.
6. **Examples**: 2-3 примера good/bad для каждого типа.

Promt-templates хранятся в `req-prompts/` директории отдельного репозитория организации с версионированием:
- `req-prompts/decompose-tz@v1.4.md`
- `req-prompts/generate-sr@v2.1.md`
- `req-prompts/generate-tc-pos-neg@v1.0.md`
- `req-prompts/critic-review-sr@v1.2.md`

`ai-provenance.prompt-template` во frontmatter артефакта содержит точную версию.

## 8. Стиль для разных моделей

Разные LLM имеют разные tendencies. Style guide должен учитывать:

| Модель | Tendency | Mitigation |
|---|---|---|
| Claude Opus | Verbose, склонен к hedging («может быть», «возможно») | Explicit constraint в prompt: «без hedging modal verbs» |
| Claude Sonnet | Конкретный, может пропустить edge case | Adversarial critic catches |
| GPT-4 | Marketing language tendency | Explicit blacklist слов из §4.2 |
| GPT-3.5 / mini | Hallucinates citations | Citation validator hook |

Style guide не запрещает использование любой модели, но требует enforcement через post-generation validation.

## 9. Validation pipeline

После генерации REQ-артефакта запускается автоматическая валидация:

```
AI generates artifact
    ↓
Style validator
    ├── citation check (each assertion has [...] or derived marker)
    ├── lexicon check (no forbidden words from §4.2)
    ├── structure check (required sections present)
    ├── length check (within bounds from §3)
    ├── canonical terms check (no forbidden synonyms)
    ↓
On fail: regenerate with refined prompt or human review
On pass: enter draft → ready transition
```

Validator — отдельный AI-агент (можно тот же что generator с другим prompt) или rule-based (regex для лексикона, parser для структуры).
